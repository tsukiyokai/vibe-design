## 设计文档审查规则(Top-10)

设计文档质量自检时，对照以下规则检查方案是否已覆盖：

1. shape维度算术是否使用int64_t? （模式1）
2. 新增枚举值/Layout时，是否列出所有需要修改的switch/if-else? （模式2）
3. 新增算子是否明确CMake + config + yaml注册清单？（模式3）
4. 从已有代码复制的部分是否逐一标注差异点？（模式4）
5. DataCopy前后是否设计了正确的PipeBarrier? （模式5）
6. TilingData是否通过共享头文件定义？（模式6）
7. workspace大小计算Host侧和Kernel侧是否一致？（模式6）
8. 内置类型成员变量是否有in-class initializer? （模式7）
9. 指针使用前是否已判空？（模式7）
10. 单次变更是否超3000行或20文件？如超出需拆分方案。（模式8）

## 高频子模式详解

以下为实战中发现的最高频三类缺陷的细分子模式，每项附典型场景和一句话审查规则。

### A. 算法正确性子模式(7项)

| 编号 | 子模式             | 典型场景                                               | 审查规则                                                 |
| ---- | ------------------ | ------------------------------------------------------ | -------------------------------------------------------- |
| A1   | 继承体系不一致     | 子类override未覆盖基类全部虚函数，遗漏的函数走默认逻辑 | 新增子类时逐一核对基类所有virtual方法是否需要override     |
| A2   | 边界条件缺失       | rankSize=1时跳过通信但未正确设置输出，count=0时未提前返回 | 对rankSize=1、count=0、单设备等退化场景逐一验证          |
| A3   | 结构体赋值遗漏     | 新增TilingData字段后Host侧赋值但Kernel侧未读取(或反之) | TilingData每次新增字段时双向检查Host赋值和Kernel读取     |
| A4   | 多阶段流程错误     | 初始化-执行-清理三阶段中跳过某阶段或顺序错乱           | 绘制状态机图，验证所有路径都经过完整阶段序列             |
| A5   | 枚举映射不完整     | 新增ReduceOp枚举但switch未覆盖、映射表未扩展           | 新增枚举值时全局搜索该enum的所有switch/if和映射表        |
| A6   | 数值精度与溢出     | shape维度用int32_t计算中间结果溢出，浮点累积误差       | 涉及shape/size/offset的算术一律使用int64_t或uint64_t     |
| A7   | 算法变体适配不全   | AllReduce支持ring但未适配halving-doubling的特殊约束     | 每种算法变体的前置条件和后置约束是否都在selector中检查   |

### B. 并发安全子模式(3项)

| 编号 | 子模式              | 典型场景                                                   | 审查规则                                                   |
| ---- | ------------------- | ---------------------------------------------------------- | ---------------------------------------------------------- |
| B1   | thread_local初始化  | thread_local变量在首次使用前未初始化，跨线程状态不隔离     | thread_local变量必须有初始值，且在线程入口处显式重置       |
| B2   | atomic读取规范      | 对atomic变量先load再判断再store，非原子的read-modify-write | atomic的read-modify-write必须用compare_exchange或fetch_add |
| B3   | 同步时序            | notify在wait之前发出导致信号丢失，barrier顺序不一致       | 绘制时序图，验证所有线程的同步点严格配对                   |

### C. 资源生命周期子模式(4项)

| 编号 | 子模式            | 典型场景                                                   | 审查规则                                                 |
| ---- | ----------------- | ---------------------------------------------------------- | -------------------------------------------------------- |
| C1   | early return跳过  | 函数提前返回跳过了后续的资源释放或状态恢复                 | 所有early return路径必须经过与正常路径相同的清理逻辑     |
| C2   | 资源标识不唯一    | 多实例场景下使用固定ID/tag导致资源冲突                     | 资源标识必须包含实例区分因子(如commIndex/rankId)         |
| C3   | 执行context错误   | 在错误的stream/context上执行操作，导致同步失效             | 验证每个操作的stream归属，特别是跨模块调用时             |
| C4   | 析构不完整        | 析构函数未释放所有成员资源，或释放顺序与构造顺序不逆序     | 析构函数逐一核对构造函数/Init中分配的每项资源是否释放   |
