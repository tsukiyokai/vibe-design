# vibe-design skill 优化设计

日期：2026-03-09

## 问题

vibe-design skill 在为 CANN/HCCL 大型 C++ 项目生成设计文档时，存在以下问题：

知识构建（根因）：
- 读得太浅，只看表面文件结构
- 读得太散，没有聚焦核心架构
- 不会主动追溯依赖关系
- 不与用户对齐理解是否正确

内容质量（继发）：
- 逐行翻译代码为中文描述
- 缺少设计决策（只写"是什么"不写"为什么"）
- 模板填充感强，章节深度均匀
- 与用户关注的重点脱节

PlantUML（继发）：
- 图表中的类名、函数名、调用关系与实际代码不符

## 方案

采用 A+B 组合：结构化探索协议 + 并行 subagent 加速。

### 1. 三层并行探索协议

替换当前模糊的"迭代阅读"，改为结构化三层探索：

- Layer 1 鸟瞰：目录结构、构建系统、模块边界 → 产出模块地图
- Layer 2 深潜：按用户指定的核心模块并行，读头文件/接口/关键数据结构 → 产出接口清单 + 符号表
- Layer 3 横切：模块间调用链、数据流、错误处理路径 → 产出交互关系

每层用 Explore subagent 执行，Layer 2 按模块并行。

### 2. 两个强制 Checkpoint

Checkpoint 1（探索后）：呈现知识摘要（模块职责、核心接口、交互关系、盲区），用户确认后才写。

Checkpoint 2（大纲阶段）：列出章节和深度分配，用户调整优先级后才写正文。

### 3. PlantUML 硬约束

- 图中元素必须来自探索阶段构建的符号表
- 调用关系必须基于 Grep 确认的实际调用链
- 绘制后逐元素回查代码自检

### 4. 写作准则

反模式（禁止）：逐行翻译代码、章节均匀用力、只写"是什么"
正模式（要求）：先结论再展开、重点章节 3x 篇幅、附代码引用、"向新人解释系统"视角

## 不改动的部分

- 模板文件（generic.md、srs-sd.md）
- PlantUML 渲染流程
- docx 转换和排版规范
